### 摘要（800字内）

本作品基于瑞芯微 RV1126B 嵌入式处理器，设计并实现了一套端侧光伏板缺陷智能检测系统。系统以 ELF-RV1126B 开发板为核心平台，搭载 Debian 12 嵌入式 Linux 操作系统，利用板载 NPU（3TOPS@INT8）高效运行 YOLOv8n 目标检测模型，实现对光伏板表面 6 类缺陷（鸟粪、灰尘、电气损伤、物理损伤、积雪及正常）的实时识别与分类，mAP50 达 0.889。

系统采用 5 线程流水线架构：V4L2 采集线程通过 USB 摄像头获取 YUYV 格式图像，经 RGA 硬件预处理转换为 RGB888 格式后送入推理队列；推理线程利用 RKNN 框架执行 INT8 量化模型推理（20-28 FPS），经 NMS 非极大值抑制、按类别置信度阈值过滤、帧间一致性消抖等多级后处理后输出检测结果；显示线程通过 DRM 帧缓冲直接渲染至 MIPI DSI 触摸屏（1024×600），叠加手写位图字体 UI；Web 服务线程提供纯 C 实现的 HTTP 服务器（端口 8080），内嵌工业控制台风格 Web 前端，支持 MJPEG 实时视频流推流、REST API 数据交互，以及手机浏览器远程访问。

系统还实现了图像预处理优化（亮度检测防黑屏、直方图均衡化改善反光场景）、二轴舵机软 PWM 步进巡检控制、Web 远程工业控制台（含登录认证、实时监控、检测记录、统计分析、系统设置四大模块，Canvas 手绘图表），以及摄像头采集异常自动恢复机制。

系统已在开发板上稳定运行，Web UI 支持多设备远程访问，具备完整的边缘 AI 视觉检测闭环。

### 第一部分 作品概述

#### 1.1 功能与特性（400字内）

本系统实现以下核心功能：

**（1）实时缺陷检测**：通过 USB 摄像头持续采集光伏板图像，利用 RKNN 推理引擎运行 YOLOv8n INT8 模型，实时识别 6 类缺陷（鸟粪、正常、灰尘、电气损伤、物理损伤、积雪），推理帧率达 20-28 FPS。

**（2）双模显示**：本地 DRM 帧缓冲 UI（1024×600 触控屏，手写位图字体）与 Web 远程工业控制台（纯 HTML/CSS/JS，无第三方依赖，响应式布局适配手机）双模并行。

**（3）Web 工业控制台**：登录认证 → 四大模块——实时监控（MJPEG 视频流+检测卡片+趋势折线图）、检测记录（可翻页表格+类别筛选+统计概览）、统计分析（Canvas 饼图/柱状图/仪表盘）、系统设置（各类别独立阈值滑块+舵机控制+摄像头开关+模型信息）。

**（4）智能后处理**：亮度检测过滤黑屏帧（mean<20 跳过推理）、高光检测过滤反光帧（饱和像素>30%跳过）、直方图均衡化改善暗光/反光场景对比度、按类别置信度阈值（鸟粪 0.55/灰尘 0.60/其余 0.50）、帧间一致性过滤（连续 3 帧确认）消抖。

**（5）舵机巡检**：二轴舵机软 PWM 控制（GPIO 直驱，SCHED_FIFO 实时线程），支持手动定位（按键 1-9 + Web 滑块）、自动步进巡检（复位→到位→采集→记录→切换），到位稳定延时 1.5s。

**（6）异常恢复**：V4L2 采集中断自动 STREAMOFF/STREAMON 恢复、MJPEG 客户端断连 SIGPIPE 保护。

#### 1.2 应用领域（400字内）

本系统面向光伏电站的端侧智能运维场景，主要应用于：

**光伏电站巡检**：通过二轴舵机带动摄像头进行步进式巡检，自动扫描光伏板阵列各区域，实时检测并记录缺陷类型与置信度。结合 Web 远程控制台，运维人员可通过手机浏览器远程查看检测画面、历史记录和统计报表，无需亲临现场即可完成日常巡检工作。

**工业视觉质检**：系统架构可迁移至零件缺陷检测、产品外观检验、包装完整性检查等工业视觉场景，仅需替换训练模型即可适配不同检测任务。

**智能安防监控**：系统的实时目标检测+远程报警能力适用于人流统计、异常行为分析、入侵检测等安防场景。

**教育科研平台**：完整的嵌入式 AI 解决方案（从模型训练、ONNX 修改、RKNN 转换到 C 代码部署的全链路），适合作为嵌入式 AI 教学实验平台。

#### 1.3 主要技术特点（400字内）

**（1）纯 C 全链路实现**：从 V4L2 采集、RGA 预处理、RKNN 推理、DRM 显示到 HTTP 服务器、JPEG 编码、JSON 序列化，全部用纯 C 语言实现，无任何高级语言运行时依赖，编译后单二进制文件直接运行于嵌入式 Linux。

**（2）cls×100 量化修复方案**：针对 RV1126B RKNN 的 INT8 量化 Bug（sigmoid 小值被零化），自主研发 ONNX 图修改方案——在输人中插入 Slice+Mul 节点将分类分数放大 100 倍，从模型层面规避了 runtime 量化精度损失。

**（3）5 线程流水线架构**：采集→推理→显示→键盘→Web 五线程松耦合，通过双队列（capQueue/dispQueue）和独立缓冲区（local_rgb）保证数据竞争安全，pthread_cond_timedwait 实现可中断等待，队列深度 4 有效缓解背压。

**（4）工业级 Web 控制台**：单文件嵌入（~28KB HTML/CSS/JS），纯 Canvas 手绘图表（饼图/柱状图/折线图/仪表盘），REST API + MJPEG 推流，预编码 JPEG 缓存实现零额外编码开销，SIGPIPE 保护防止客户端断连崩溃。

**（5）软硬件协同**：RGA 硬件加速格式转换/缩放、直接 DRM 帧缓冲渲染（绕开 X11/Wayland 开销）、GPIO 软件 PWM 生成 50Hz 舵机波形（SCHED_FIFO 实时优先级防抖）、V4L2 异常自动恢复。

#### 1.5 主要创新点（200字内）

1. **cls×100 ONNX 图修改方案**：从模型层面解决 RKNN INT8 量化 sigmoid 小值精度损失问题，比 runtime 侧 workaround 更彻底、更通用。
2. **嵌入式纯 C Web 服务**：不依赖任何框架，单文件 `web_ui.h`（~1000 行）实现 HTTP 服务器 + MJPEG 推流 + JPEG 编码 + JSON API + Canvas 图表工业控制台。

4. **反光场景端侧图像增强**：高光检测自动跳过 + RGB 三通道直方图均衡化，在嵌入式端实时实现工业图像预处理。
5. **软硬件全链路闭环**：从芯片 PWM/GPIO 底层驱动到 Web 控制台的一体化设计，代码体积小（~2000 行 C）但功能完整。

### 第二部分 系统组成及功能说明

#### 2.1 整体介绍

**系统整体框图**：

```
┌──────────────────────────────────────────────────────┐
│                  RV1126B (ELF-RV1126B)                │
│                                                      │
│  ┌─────────┐  ┌────────┐  ┌────────┐  ┌──────────┐  │
│  │USB摄像头│→│ V4L2   │→│  RGA   │→│  RKNN    │  │
│  │640x480  │  │采集线程│  │预处理  │  │推理线程  │  │
│  └─────────┘  └────┬───┘  └────────┘  └────┬─────┘  │
│                    │    capQueue(4)        │         │
│                    └──────────────────────┘         │
│                                                    │
│  ┌──────────┐  ┌──────────┐  ┌──────────────────┐  │
│  │  DRM显示 │←│显示线程   │  │   Web线程(8080)  │  │
│  │1024x600  │  │后缓冲渲染 │  │HTTP+MJPEG+JSON   │  │
│  └──────────┘  └──────────┘  └────────┬─────────┘  │
│                                       │             │
│  ┌──────────┐  ┌──────────┐   上位机浏览器/手机     │
│  │ 键盘输入 │  │舵机控制   │   http://IP:8080       │
│  │S/+/-/R/Q │  │GPIO软PWM  │                        │
│  └──────────┘  └──────────┘                        │
└──────────────────────────────────────────────────────┘
```

系统以 RV1126B 为主控，5 个线程协同工作：采集线程通过 V4L2 接口从 USB 摄像头获取 YUYV 格式图像帧，经 RGA 硬件模块进行 YUYV→RGB888 格式转换和 640×640 resize，将预处理后的帧推入 capQueue 队列（深度 4）。推理线程从 capQueue 取出帧，先进行图像预处理（亮度检测/高光检测/直方图均衡化），然后通过 RKNN 框架在 NPU 上运行 YOLOv8n INT8 模型推理，经 NMS、按类别阈值过滤、帧间一致性消抖后，推送检测结果到 dispQueue 队列供显示线程渲染，同时 JPEG 预编码后发布到 Web 线程。显示线程通过 DRM 直接写帧缓冲在 MIPI 屏幕上渲染摄像头画面、检测框和手写 UI。输入线程读取键盘按键控制摄像头开关、阈值调节和舵机操作。Web 线程独立监听 8080 端口，提供 HTTP 服务、MJPEG 视频流推送和 REST API 数据交互。

#### 2.2 硬件系统介绍

**2.2.1 硬件整体介绍**

| 组件   | 型号/规格                 | 用途                    |
| ------ | ------------------------- | ----------------------- |
| 核心板 | ELF-RV1126B (RV1126B SoC) | 主控+NPU推理            |
| 摄像头 | icspring USB Camera       | 图像采集 640×480@30fps |
| 显示屏 | 7寸 MIPI DSI 1024×600    | 本地实时显示            |
| 舵机   | SG90 ×2                  | 二轴摄像头巡检          |
| 电源   | 5V/3A 外接电源            | 舵机独立供电            |
| 网络   | WiFi (RTL8821CU)          | Web 远程访问            |

**2.2.2 机械设计介绍**

采用二轴云台结构：X 轴舵机（水平旋转 0-180°）负责摄像头指向不同的光伏板列，Y 轴舵机（俯仰 0-90°）负责调整摄像头俯角以适应不同安装高度。舵机支架通过 3D 打印件/铝合金支架连接，摄像头固定于 Y 轴舵机转盘上。两个舵机由独立 5V/3A 外接电源供电，GND 与开发板共地以保证 PWM 信号参考电压一致。

**2.2.3 电路各模块介绍**

舵机控制信号由开发板 P16 排针输出：

- X 轴舵机信号：GPIO128（GPIO4_A0, P16-29 脚）
- Y 轴舵机信号：GPIO23（GPIO0_C7, P16-33 脚）
- 两个舵机 GND 接开发板 GND，VCC 接外部 5V 电源
- 舵机信号电压 3.3V（RV1126B GPIO 电平），经验证可驱动 SG90 舵机

USB 摄像头直接插入开发板 USB3.0 接口，设备节点为 /dev/video52。

#### 2.3 软件系统介绍

**2.3.1 软件整体介绍**

软件采用纯 C 语言实现，单文件 `solar_defect.c`（主程序 ~1050 行）+ `web_ui.h`（Web 前端 ~1050 行）+ `servo.h`（舵机驱动 ~200 行），无任何外部运行时依赖。

编译命令：

```bash
gcc solar_defect.c -o solar_defect -lrknnrt -lrga -ldrm -ldl -lm -lpthread -ljpeg
```

系统运行于 Debian 12 嵌入式 Linux，通过动态加载 `librknnrt.so` 访问 NPU 推理引擎，通过 `librga.so` 实现硬件加速图像格式转换。

**2.3.2 软件各模块介绍**

**采集线程**：V4L2 MMAP 模式 4 缓冲区轮转 → select() 2 秒超时等待 → VIDIOC_DQBUF 取帧 → RGA improcess YUYV→RGB888 + 640×640 resize → VIDIOC_QBUF 还帧 → 推入 capQueue。异常恢复机制：连续 DQBUF 超时 >30 次（~60秒）触发 STREAMOFF→延时→STREAMON 自动恢复。

**推理线程**：从 capQueue 取帧 → memcpy 到 local_rgb（独立缓冲防竞态）→ 亮度检测（mean<20 跳过）→ 高光检测（饱和>30%跳过）→ RGB 三通道直方图均衡化 → 摄像头开关检查 → RKNN 推理 → 后处理（bbox 解析、cls/100 还原、按类别阈值过滤、NMS IoU 0.45、最小面积 400px² 过滤、帧间一致性 IoU 0.20 连续 3 帧确认）→ JPEG 预编码（锁外编码，不阻塞 API）→ 推入 dispQueue。

**显示线程**：从 dispQueue 取帧 → 后缓冲渲染（背景填充、摄像头画面 16.16 定点缩放叠加、检测框 3px 绘制、UI 层叠加——状态栏+按键提示+日志面板）→ memcpy 到 DRM 映射内存。

**Web 线程**：socket bind 8080 → select 监听 → accept → 解析 HTTP 方法+路径 → 路由分发（GET: /, /stream, /stats, /records, /servo; POST: /capture, /threshold, /servo）→ MJPEG 流使用预编码 JPEG 缓存（所有连接共享，零实时编码开销）→ O_NONBLOCK 写防止客户端阻塞 → SIGPIPE 忽略防止断连崩溃。

**舵机线程**：SCHED_FIFO 优先级 90 实时线程 → 50Hz 周期脉冲生成 → 两轴同步 HIGH → 按各轴脉宽顺序 LOW → 剩余时间补齐 20ms → lseek+write 持久 fd 直接写 GPIO value 文件。

### 第四部分 总结

#### 4.1 可扩展之处（300字内）

1. **模型升级**：可替换为 YOLOv8s 或更高精度模型，或采用 QAT（量化感知训练）进一步减少 INT8 量化精度损失。
2. **数据集增强**：针对反光、逆光、遮挡等实际复杂场景补充训练数据，提升模型鲁棒性。
3. **检测记录持久化**：增加 SQLite 数据库或文件系统存储，实现检测记录的长期保存和离线查询。
4. **实时告警推送**：通过 WebSocket 实现缺陷检出时的即时推送通知。
5. **多摄像头支持**：利用 RV1126B 双 MIPI CSI 接口，接入多个摄像头实现更大范围覆盖。
6. **4G 远程上报**：利用 4G 模块实现检测结果上传云端，支持远程运维和集中管理。
7. **舵机轨迹优化**：加入梯形加减速控制，减少舵机到位过冲和机械抖动。
8. **硬件 PWM 迁移**：使用 PCA9685 等 I2C 舵机驱动板替代软件 PWM，提升波形精度。
9. **模型在线更新**：通过 Web 接口上传新模型文件，无需重新编译即可切换。

#### 4.2 心得体会（1000字内）

**项目开发历程回顾**：

本项目历时约两个月，经历了从原型验证到系统集成的完整开发流程。最初阶段，我们通过编写 test_v4l2.c、test_rga.c、test_drm.c、test_rknn_infer.c 等独立测试程序，逐一验证了 V4L2 采集、RGA 预处理、DRM 显示、RKNN 推理等核心模块的功能，为后续集成奠定了坚实的基础。

**关键技术难点与解决**：

最大的技术挑战是 RV1126B 的 RKNN runtime v2.3.2 存在 INT8 量化 Bug——YOLO 模型输出的 sigmoid 分类分数（0-0.9 范围）经过 INT8 量化后全部变成 0，导致模型完全无法识别任何目标。我们尝试了 FP16 不量化、输出分离（3 个 output）等多种方案均告失败，最终通过分析 RKNN 量化原理，创造性地提出了"cls×100 放大"方案：在 ONNX 模型输出端插入 Slice+Mul 节点，将分类分数放大 100 倍（使得量化前值落在 0-100 范围，INT8 可表示），在 C 代码读取时除以 100 还原。这个方案从模型层面规避了量化精度损失，且不需要修改 RKNN runtime 本身。

第二个重要挑战是本地 `rknn_api.h` 头文件与开发板实际 ABI 不匹配，导致运行时 `rknn_output.size` 读取为错误值（127 而非正确的 336000），引发段错误。通过对照板载 `/usr/include/rknn_api.h` 逐字段核对结构体布局（发现 `rknn_output` 的 `want_float`/`is_prealloc` 字段类型从 `unsigned int` 改为 `uint8_t`，且字段顺序有变化），完成了 ABI 对齐修复。

**软 PWM 舵机控制的波折**：

在舵机控制环节，原本计划使用硬件 PWM（/sys/class/pwm/），但发现板子全部 3 路已启用 PWM 都被系统关键功能（CPU 供电 vdd-cpu、屏幕背光 backlight-dsi、LCD 背光 backlight-lcd）占用，而设备树中其余 20+ 路 PWM 处于 disabled 状态且无法通过 configfs 动态启用。最终采用了软件 PWM 方案——通过 GPIO sysfs 直接输出 50Hz 波形，并针对 sysfs fopen/fclose 开销过大的问题，优化为持久文件描述符 + lseek+write 方式，将每次 IO 操作从数百微秒降至数十微秒；通过设置 SCHED_FIFO 实时调度优先级（priority=90）和 1° 死区过滤，有效消除了线程抢占导致的波形抖动。GPIO 引脚选择也经历了多次尝试，最终确定使用 GPIO128（GPIO4_A0, P16-29 脚）和 GPIO23（GPIO0_C7, P16-33 脚）作为舵机控制信号输出。

**反光场景优化经验**：

实际测试中发现光伏板表面反光严重影响检测准确性——强反光区域被模型误判为缺陷。我们在软件层面实现了双重应对：首先加入高光检测（统计 RGB 三通道均超过 240 的饱和像素，占比>30% 则跳过推理），过滤极端反光帧；其次在推理前进行 RGB 三通道独立直方图均衡化，改善反光/暗光场景下的图像对比度，使模型能更好地提取缺陷特征。两项优化仅增加了约 1-2ms 的预处理开销，不影响系统实时性。

**Web 工业控制台的迭代**：

从最初的简单 HTML 页面（3 个按钮 + 1 个表格），到当前的四模块工业控制台（登录认证 + Canvas 手绘图表 + MJPEG 流 + REST API），经历了三次大的重构。第一次重构加入了 MJPEG 视频流和中文界面；第二次重构实现了完整的 SPA 架构（侧边栏导航 + 4 个功能页面 + 各类别阈值滑块）；第三次重构针对 MJPEG 流卡顿问题，将 JPEG 编码从互斥锁内移到锁外，并增加了 SIGPIPE 保护防止客户端断连导致进程崩溃。

**对嵌入式和边缘 AI 的认知提升**：

通过本项目，我们深刻体会到嵌入式 AI 开发与服务器端 AI 开发的本质区别：不是"把 Python 代码搬到板子上跑"，而是需要从底层硬件接口（V4L2/DRM/PWM/GPIO）到上层应用（HTTP/MJPEG/Canvas 图表）的全栈掌控能力。每一行代码都与硬件紧密相关，每一个性能指标都受到芯片算力、内存带宽、外设接口的严格约束。这种"螺丝壳里做道场"的工程体验，让我们对嵌入式系统设计有了远超课堂教材的理解。
